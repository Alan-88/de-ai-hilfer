任务：根据词典 facts 和 lexical branch skeleton，生成德语词条的学习内容模块 Markdown。

你会收到：
- 查询词
- Dictionary Facts JSON（词典事实）
- Branch Skeleton（canonical branch / meanings / grammar 骨架）

输出范围：
1. 只生成三个学习内容模块：`应用与例句`、`词汇网络`、`深度解析与避坑`。
2. 不生成 headword、IPA、核心释义、语法详情或完整词条头部。
3. Branch Skeleton 是本轮生成的边界输入；不要推翻 branch 边界，不要随意新增或删除 branch。
4. 如果输入事实彼此明显自相矛盾，只在相关模块中谨慎说明，不要自行重建 skeleton。

内容原则：
1. “应用与例句”不要机械输出 2-3 条，而要自适应地覆盖高价值 usage units。
2. 高价值 usage unit 指：高频固定搭配、核心句型结构、学习者容易混淆但真实使用里常见的基本用法、能明显帮助区分 branch/语气/搭配限制/语义重心的典型场景。
3. 不要为了凑数量输出低信息密度模块；也不要因为“通常 2-3 条”而漏掉关键用法。
4. 如果某个词几乎没有固定搭配，也可以只给少量真正有价值的基本用法模块。
5. `usage module` 必须是“可直接拿来识别、复现或造句的使用单元”。语法规则、重音规律、构词解释、主宾逻辑说明或 branch 元信息，应写到“深度解析与避坑”。
6. 不要为了照顾每个 branch 都露面，就牺牲更高价值的固定表达或核心用法。Branch Skeleton 是边界，不是 `usage_modules` 的强制配额来源。
7. 如果主 branch 本身已经能提供 3 条以上高价值 usage modules，默认不要再为了“完整性”去展开次级 branch。次级 branch 只有在本身极常见、极高频，或拥有非常值得单独记忆的固定表达时，才值得占用 usage module。
8. 文学性、书面性、军事语境或边缘义项，即使作为合法 secondary branch 存在，也默认优先留给释义/语法层；只有确实有理解价值时，才在“深度解析与避坑”中简要说明。
9. “深度解析与避坑”默认走“理解型解析”，优先回答：为什么这个词会这样用、脑中该如何建立判断模型、它和近邻词/近邻结构差在哪。
10. `deep_insights` 优先保留 2-4 条真正有解释价值的内容：核心理解、辨析/边界、确实有助于记忆的词源/构词逻辑、高价值常见误区。
11. 不要把 `deep_insights` 写成纯规则清单，也不要机械复述 Branch Skeleton 已展示的事实。
12. 若词典事实里有明确词源材料，且确实有助于理解或记忆，尽量保留“词源”小节；否则不要机械硬塞。
13. 涉及德语格位时，术语必须准确：`Nominativ=第一格/主格`，`Genitiv=第二格/属格`，`Dativ=第三格/与格`，`Akkusativ=第四格/宾格`。
14. `usage_modules` 标题必须是自然德语搭配、固定表达或可复用句型；不要为了概括而写不自然标题。
15. 每条 `例句翻译` 必须逐句忠实对应 `场景例句`。

Markdown 输出格式：
1. 直接输出 Markdown，不要输出 JSON、代码块包裹或额外说明。
2. 只能使用下面三个一级结构，标题名保持一致：
   - `#### 应用与例句 (Anwendung & Beispiele)`
   - `#### 词汇网络 (Wortnetz)`
   - `#### 深度解析与避坑 (Einblicke)`
3. 排除范围：以下内容由其他结构层承接，本模块不得重复生成：
   - `### {headword}`
   - `/IPA/`
   - `#### 核心释义 (Bedeutung)`
   - `#### 语法详情 (Grammatik)`
   - Markdown 表格形式的语法信息
4. “应用与例句”中的每个 usage module 必须使用稳定四行格式；模块之间空一行：
   德语搭配 / 句型结构
   用法解析：1-3 句中文解释，重点写什么时候用、为什么这样用、和什么容易混淆
   场景例句：一条自然、完整的德语例句
   例句翻译：对应中文翻译
5. “词汇网络”按三类分组：词族、同义词、反义词。每个条目必须使用下面格式，不要省略方括号：
   - [词] (*词性*, *中文释义*, *英文释义*)
   必须显式写出三个分组标题：`**词族**`、`**同义词**`、`**反义词**`。
   如果某一组没有高价值内容，写 `*   无高价值项`，不要把所有相关词混成一个扁平列表。
6. “深度解析与避坑”中的每个小节前用列表项标题，例如：
   * 多维解释
     这里写正文……
   * 词源
     这里写正文……
   正文优先写成 1-2 段自然解释；只有确实需要并列对比时才使用短列表。

质量要求：
1. 不要把 Branch Skeleton JSON 原样贴出来。
2. 不要把 branch 组织逻辑误写成 usage 模块本身。
3. 不要重新生成 Branch Skeleton 已提供的释义、音标或语法表；如需引用语法事实，只能在用法解释或深度解析里自然提到。
4. 不要为了整齐牺牲信息量；也不要为了丰富而胡乱扩写。
5. 如果某一部分证据不足，就写少一点，但不要写空洞套话。
